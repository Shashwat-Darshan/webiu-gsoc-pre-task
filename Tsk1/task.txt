Task 1 — Design
Designing a Scalable GitHub Data Aggregation System
Objective
Design an architecture for a system that efficiently aggregates repository data from GitHub and serves it to a website while minimizing API usage and ensuring scalability.

Scenario
An organization maintains 300+ GitHub repositories whose project information must be displayed on a website. The system should automatically update repository data when changes occur.

Requirements
Architecture Design
Propose a system architecture that collects, processes, stores, and serves repository data efficiently.

Core Components
Your design must include data ingestion, processing layer, storage layer, API layer, and caching mechanism.

Rate Limit Handling
Explain strategies to minimize GitHub API usage and remain within API rate limits.

Update Mechanism
Describe how the system detects repository updates (e.g., webhooks, scheduled jobs, incremental updates).

Data Storage Strategy
Specify what repository data should be stored persistently and what should be fetched dynamically.

Scalability Plan
Explain how the architecture scales if repositories increase from 300 to 10,000.

Performance Optimization
Discuss caching strategies and techniques to ensure fast response times for frontend requests.

Failure Handling
Explain how the system handles API failures, rate-limit exhaustion, or unavailable repositories.

API Flow
Describe how the frontend interacts with the backend to retrieve repository data.

Technology Choices
Justify the technologies, frameworks, and infrastructure components you would use.

Deliverables
Architecture diagram
1–2 page design explanation
API flow description
Technology justification
